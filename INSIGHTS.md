# INSIGHTS.md — Three Things We Learned About LILA BLACK

Generated by exploring the full 5-day dataset (89,104 events, 796 matches, 339 players)
with the visualization tool, then verified directly against the parquet data with pandas.

---

## 1. Matches are effectively solo sessions with a bot fill, not human PvP

**What caught our eye:** The heatmaps and player-count filter made it obvious almost
every match on screen had exactly one blue (human) path and a swarm of orange (bot)
paths.

**The numbers:** Across 780 matches with at least one human player, **779 have exactly
one human player** and only 1 match has two. Mean humans-per-match = 1.001. At the
event level, human-vs-human combat is nearly nonexistent: **3 `Kill` events** (human
killed human) versus **2,415 `BotKill` events** (human killed bot) — bot kills outnumber
PvP kills 805 to 1. `Killed` (died to another human) is also just 3 events, vs. 700
`BotKilled`.

**What this affects / actionable items:**
- Any PvP-balance metric (time-to-kill, weapon pick rates in human fights, contested
  loot zones) **cannot be measured from this dataset** — there isn't enough human-human
  combat to be statistically meaningful (3 events total).
- If this is meant to represent live production traffic, worth checking whether
  matchmaking is placing humans together at all, or whether this 5-day window happens
  to be dominated by solo QA/playtest sessions rather than real concurrent player
  activity.
- If it's true production data, the actionable question shifts: is the human/bot fill
  ratio (avg. ~30 bots per human, based on match rosters) tuned correctly? A player who
  never meets another human may churn faster than one who does — worth checking
  retention against number of human encounters per match.

**Why a level designer should care:** Combat-zone and death-zone heatmaps built from
this data are really "where do humans fight bots" maps, not "where do humans fight each
other" maps. Any redesign decision aimed at improving PvP hotspots needs a different,
human-dense dataset first — otherwise you're optimizing space for an encounter type that
essentially isn't happening.

## 2. AmbroseValley absorbs 68% of matches while GrandRift — the richer loot map — gets 7.7%

**What caught our eye:** Switching the map filter, GrandRift's match dropdown was
strikingly short compared to AmbroseValley's.

**The numbers:** Match share by map: **AmbroseValley 68.5%** (566 matches),
**Lockdown 23.8%** (171), **GrandRift 7.7%** (59). Yet GrandRift has the *highest* loot
density per match at **14.9 loot events/match**, vs. AmbroseValley's 17.6 (close) and
Lockdown's 12.0 — so GrandRift isn't a sparse or unfinished map; it has comparable
content density to the most-played map but a fraction of the traffic.

**What this affects / actionable items:**
- Check the map-rotation/matchmaking weights — if GrandRift is supposed to be in equal
  rotation, an 8.9:1 skew toward AmbroseValley suggests a weighting bug or a rotation
  gate (level requirement, playlist toggle) is unintentionally suppressing it.
- Metrics affected: overall loot-pacing tuning, storm-timer tuning, and matchmaking
  queue-time targets are all currently being validated almost entirely on AmbroseValley
  data. GrandRift-specific balance issues (queue starvation aside) would be invisible
  until the map gets real traffic.
- If the low rotation is intentional (e.g., GrandRift is a limited-time or beta map),
  this is still useful as a baseline to compare against once rotation opens up.

**Why a level designer should care:** Design iteration velocity on GrandRift is
bottlenecked by sample size — with only 59 matches, per-zone kill/death/loot heatmaps
for that map are noisy and shouldn't be over-interpreted yet. Fixing the rotation
(if it's a bug) is a prerequisite for trusting any GrandRift-specific redesign decision.

## 3. Storm death rate is 3x higher on Lockdown and GrandRift than on AmbroseValley

**What caught our eye:** Toggling the "Storm deaths" heatmap layer per map, the marker
density looked visibly sparser on AmbroseValley relative to how often it's played.

**The numbers:** Storm-death rate (storm deaths ÷ human player-match sessions on that
map): **AmbroseValley 3.1%** (17 storm deaths / 554 sessions), **GrandRift 8.8%**
(5/57), **Lockdown 10.0%** (17/170). Lockdown and GrandRift players die to the storm at
roughly **3x the rate** of AmbroseValley players, despite the storm mechanic presumably
being tuned the same way across maps.

**What this affects / actionable items:**
- Metrics affected: extraction success rate and average match length on the two smaller
  maps are likely being suppressed by storm deaths that feel unfair rather than
  earned — worth checking player sentiment/complaint tags for Lockdown and GrandRift
  specifically.
- Actionable item: review whether the storm shrink timer and safe-zone travel distance
  are scaled to each map's physical size (Lockdown is explicitly the "smaller/close-
  quarters map" per the README) — a fixed global storm timer would punish smaller maps
  disproportionately, which is consistent with what we're seeing.
- A/B test candidate: slow the storm shrink rate specifically on Lockdown/GrandRift (or
  extend the initial safe-zone radius) and re-measure storm-death rate against the
  3.1% AmbroseValley baseline established here.

**Why a level designer should care:** This is a direct, quantified signal that storm
pacing — a core system, not just level geometry — may need to be per-map rather than
global. It also gives a concrete baseline (3.1%) to test against, rather than tuning
storm timers on intuition alone.

---

### Caveats
- February 14 is a partial day, so day-level (not map-level) comparisons should account
  for its lower row count.
- A small number of bot rows (751) carry non-standard event names (e.g. `Loot`/`Position`
  instead of `BotLoot`/`BotPosition`); they were classified by `user_id` shape, not event
  name, so they're included correctly in bot-side stats — but if the game team's own
  analytics pipeline classifies by event name instead, its bot/human split may disagree
  slightly with the numbers above.
